# الأمان والصلاحيات

## نموذج التهديد

الخطر الرئيسي هو وصول مستخدم منشأة A إلى بيانات منشأة B عبر تغيير `organization_id` أو معرف سجل في الطلب. لذلك لا تعتمد الحماية على إخفاء الروابط أو مرشحات الواجهة؛ تطبق داخل PostgreSQL عبر RLS.

## المصادقة وحماية المسارات

- كلمات المرور والجلسات تديرها Supabase Auth ولا تحفظ في جداول التطبيق.
- Proxy يجدد كوكيز المصادقة ويحوّل الزائر غير المسجل عن المسارات المحمية.
- كل صفحة محمية تعيد التحقق من المستخدم على الخادم بواسطة `auth.getUser()`.
- قيمة المنشأة النشطة لا تُوثق من Cookie وحدها؛ يجب أن تطابق عضوية ظاهرة للمستخدم.
- عمليات إنشاء المنشأة وتبديلها والتحقق من القدرة خادمية، وتظل RLS هي طبقة المنع النهائية.
- رابط تأكيد التسجيل يبنى من `NEXT_PUBLIC_SITE_URL` الموثوق، ولا يأخذ أصله من ترويسة طلب قابلة للتلاعب.
- معامل `next` يقبل مسارًا نسبيًا من نفس الأصل فقط، ويرفض الشرطات العكسية والمحارف الضابطة والترميز المتكرر لمسار خارجي.
- الردود تحمل سياسة أمان للمحتوى، ومنع تضمين الإطارات، ومنع تخمين نوع المحتوى، وسياسة إحالة وصلاحيات متصفح مقيدة. المسارات المحمية تحمل كذلك `private, no-store`.

## أدوار التطبيق

| العملية | owner | manager | accountant | viewer |
| --- | ---: | ---: | ---: | ---: |
| قراءة بيانات التشغيل والمؤشرات | نعم | نعم | نعم | نعم |
| إدارة العقارات والوحدات والمستأجرين | نعم | نعم | لا | لا |
| إنشاء وتعديل العقود الأساسية | نعم | نعم | مالي محدود فقط | لا |
| إضافة/تعديل الدفعات والمصروفات | نعم | نعم | نعم | لا |
| إلغاء دفعة أو حذف مصروف منطقيًا | نعم | نعم | لا | لا |
| تعديل Benchmarks | نعم | نعم | لا | لا |
| حذف Benchmark | نعم | لا | لا | لا |
| إنشاء Snapshot | نعم | نعم | لا | لا |
| إدارة الأعضاء غير المالك الأساسي | نعم | نعم دون منح owner | لا | لا |
| حذف البيانات الرئيسية حذفًا صريحًا | نعم | لا | لا | لا |
| حذف المنشأة | نعم | لا | لا | لا |
| تغيير/إزالة المالك الأساسي مباشرة | لا | لا | لا | لا |

المالك الأساسي لا يغير مباشرة لحماية المنشأة من القفل أو الاستيلاء. نقل الملكية، إن أضيف مستقبلًا، يجب أن يكون إجراءً آمنًا منفصلًا ومراجَعًا ومدققًا.

## دوال RLS

- `is_org_member(target_organization_id uuid)`: تتحقق من عضوية `auth.uid()`.
- `has_org_role(target_organization_id uuid, allowed_roles text[])`: تتحقق من العضوية وأن الدور ضمن القائمة.

الدالتان `security definer` مع `set search_path = ''` لمنع حقن كائن عبر مسار البحث ولتفادي عودية سياسات `organization_members`. التنفيذ ممنوح للمستخدمين المسجلين فقط.

## نمط السياسات

- القراءة: عضوية صحيحة غالبًا.
- الإدخال والتعديل: عضوية + دور مسموح.
- `with check` يعيد التحقق من السجل الجديد، وليس السجل القديم فقط.
- حذف الجداول الرئيسية الصريح محصور بالمالك؛ manager يستخدم التدفقات الطبيعية والحذف المنطقي.
- حذف المنشأة كاملة ينفذ عبر `delete_organization()` التي تتحقق من دور owner وتحذف السجلات التابعة بترتيب آمن داخل معاملة واحدة.
- الإلغاء أو الحذف المنطقي للدفعات والمصروفات مسموح للمالك والمدير، بينما يمنع الحذف المادي عبر واجهة قاعدة البيانات حتى يبقى السجل قابلًا للتدقيق.
- `audit_logs` لا يقبل إدخالًا أو تعديلًا مباشرًا من مستخدم التطبيق.
- Snapshots لا تقبل تعديلًا؛ الإنشاء للمالك والمدير والحذف للمالك فقط.
- إجراءات الوحدة والعقد المركبة تستخدم دوال `security definer` محدودة التنفيذ وتعيد فحص دور المالك أو المدير داخل المعاملة نفسها.
- `record_schedule_payment()` يعيد فحص دور المالك أو المدير أو المحاسب، ويقفل الاستحقاق قبل حساب المتبقي لمنع التحصيل الزائد المتزامن.
- `void_payment()` محصور بالمالك والمدير ويحوّل الدفعة إلى حذف منطقي، ثم يعاد حساب حالة الاستحقاق تلقائيًا.

## قيود لا تكفيها RLS

RLS تتحكم في الصف ولا تميز دائمًا أي عمود تغير. لذلك تستخدم المشغلات أيضًا:

- منع تعديل `organization_id` في أي سجل تابع.
- منع تعديل `created_by` للسجلات المالية ومراجع السوق.
- تقييد accountant في `leases` على `annual_rent` و`security_deposit` و`grace_period_days` فقط.
- منع تغيير `organizations.created_by`.
- منع خفض دور المالك الأساسي أو حذف عضويته مباشرة.
- منع ربط دفعة بعقد مختلف عن عقد الاستحقاق، ومنع أي مجموع دفعات يتجاوز المبلغ المستحق.
- منع أكثر من Snapshot للعقار في يوم واحد مع الإبقاء على اللقطات غير قابلة للتعديل.
- منع تاريخ دفع مستقبلي، ومنع اعتبار مصروف مستقبلي مدفوعًا، ومنع مرجع سوق انتهت فترته في المستقبل، وفق تاريخ `Asia/Riyadh`.
- منع إنشاء دفعة أو مصروف محذوف منطقيًا مسبقًا، وليس فقط منع التحايل عليه في التعديل.

## التدقيق

تسجل مشغلات قاعدة البيانات عمليات INSERT وUPDATE وDELETE على:

- `payments`.
- `expenses`.
- `leases`.
- `market_benchmarks`.

كما تسجل تغيير `units.current_annual_rent`. يحتوي السجل على المنشأة والمستخدم والعملية والجدول ومعرف السجل والقيم القديمة والجديدة. قد يكون `user_id` فارغًا لعملية نظام موثوقة بلا جلسة مستخدم، لكن عمليات التطبيق العادية تحمل `auth.uid()`.

يسجل جدول `unit_status_periods` كذلك تغييرات حالة الوحدة. دوال المرحلة الخامسة لا تمنح المستخدم وصولًا مباشرًا أوسع؛ إذ أُلغي التنفيذ العام، ومُنح للمستخدم المسجل فقط بعد تحقق الدور داخل الدالة.

## الأسرار والبيانات الشخصية

- لا تخزن مفاتيح في الكود أو SQL أو Seed Data.
- مفتاح `service_role` للخادم فقط ولا يظهر للمتصفح.
- لا تخزن صورة الهوية. `national_id_reference` ليس حقلًا لرقم الهوية الخام؛ يستخدم مرجعًا خارجيًا أو قيمة مموهة عند الحاجة.
- لوحة المحفظة لا تعرض تفاصيل المستأجرين غير اللازمة.
- الاستعلامات الحساسة تحدد الأعمدة ولا تستخدم `select *`.

## قائمة مراجعة لكل جدول جديد

1. هل يحتاج `organization_id not null`؟
2. هل العلاقة المركبة تمنع العبور بين المنشآت؟
3. هل فُعل RLS؟
4. هل توجد سياسات منفصلة لكل عملية مطلوبة؟
5. هل الدور، لا العضوية فقط، مفروض على الكتابة؟
6. هل تغيير المنشأة ممنوع؟
7. هل يحتاج حذفًا منطقيًا أو تدقيقًا؟
8. هل اختبارات العزل والدور تغطيه؟

## مراجعة الاعتماديات

يشغّل المشروع `npm run audit:prod` لمراجعة اعتماديات الإنتاج. نتيجة المهمة السابعة لا تحتوي ثغرات عالية أو حرجة. توجد ملاحظتان متوسطتان في `postcss` المضمّن عبر أداة البناء، ولا يتوفر لهما إصلاح آلي حاليًا. التطبيق لا يستقبل CSS من المستخدم، لذلك وثقتا كمخاطرة منخفضة التعرض تحتاج المتابعة عند تحديث Next.js وسلسلة البناء. التفاصيل في [stage-seven-review.md](stage-seven-review.md).
